What's inside the JWT
HEADER

PAYLOAD
{ "id": "<MongoDB _id>", "username": "john", "role": "user/admin", "iat": 1234567, "exp": 1235167 }
id used to scope DB queries. role used by adminGuard. exp = 7 days from issue.
SIGNATURE
HMACSHA256(base64(header)+"."+base64(payload), "mysecretkey")
Verified by jwt.verify() on every protected request. Tampered tokens fail here.

authMiddleware internals
authMiddleware — how it works

Reads req.headers["authorization"]. Splits on space to get the token after "Bearer". Calls jwt.verify(token, SECRET) — validates signature + checks expiry. On success attaches decoded payload as req.user.

const token = authHeader.split(" ")[1];
jwt.verify(token, SECRET, (err, user) => {
  if (err) return res.status(403).json(...);
  req.user = user; // {id, username, role}
  next();
});
// Applied on: /urldata /labels /utm /org
adminMiddleware internals
adminMiddleware — how it works

Does everything authMiddleware does, then additionally checks user.role !== "admin" and returns 403 if not admin. Attached at the app level for /admin routes.

jwt.verify(token, SECRET, (err, user) => {
  if (err) return res.status(403).json(...);
  if (user.role !== "admin")
    return res.status(403).json(...);
  req.user = user;
  next();
});
// Applied on: /admin routes
Failure cases
No token → 401

If Authorization header is missing, middleware returns 401 immediately without calling next().

Invalid / expired token → 403

jwt.verify() errors if tampered or expired. Returns 403. Angular Observable errors — component doesn't update.

parseVisit — detection logic
parseVisit utility

Reads User-Agent header. Uses ordered regex tests. Browser order matters — Edge/Opera must be checked before Chrome since their UA strings also contain "Chrome".

// Device (checked in order)
/iphone|ipad|ipod/i → "iOS"
/android/i → "Android"
/tablet/i → "Tablet"
default → "Desktop"

// Browser (order matters!)
/edg\//i → "Edge"
/opr\//i → "Opera"
/chrome/i → "Chrome"
/firefox/i → "Firefox"
/safari/i → "Safari"

buildParams — linkData injection
Query param injection

linkData[] key-value pairs are converted to query string and appended to destination URL. Uses & if URL already has ?, else uses ?. For intent:// URLs the params are injected into the path part before the #Intent separator.

linkData = [{key:"offer", value:"SAVE20"}]
→ "https://site.com/page?offer=SAVE20"

Charts rendered on the page
Visit Trend Chart (line chart)

Built by buildTrendChart(). Uses daily/weekly/monthly arrays from the response. Tab buttons switch the active dataset. X-axis = date labels, Y-axis = visit count. Chart.js line chart with fill area, tension 0.4, blue color scheme. Old chart is destroyed before building new one to prevent canvas reuse errors.

Breakdown Chart (line chart)

Built by buildBreakdownChart(). Tab buttons switch between device / browser / source / medium / campaign / referrers. Each tab uses a different array from the response (devices[], browsers[], utmBySource[], etc.). Same Chart.js line style as trend chart for visual consistency.

UTM Performance Table

Rendered from analyticsData.utmBreakdown[]. Shows each UTM link's name, source, medium, campaign, click count, and a percentage bar. Bar width = (utm.clicks / totalUtmClicks) * 100. Only shown if utmBreakdown.length > 0.

Date preset buttons

7d / 30d / 90d buttons call setPreset(). Sets fromDate and toDate, then if analyticsData already exists, calls analytics() again to refetch with the new range. "Custom" preset lets user pick exact dates via date inputs.

Stat cards (4 summary cards)

Show filteredVisits (total in range), androidClicks, desktopClicks, and active UTM count. These come directly from the response fields — no client-side computation needed.

How trend data is built on backend
Daily / Weekly / Monthly maps

Backend pre-fills maps with zeroes for every slot in the date range (up to 90 days back). Then iterates filteredHistory[] and increments the matching slot for each visit. Ensures gaps with zero visits show as 0 on the chart rather than being missing.

// Pre-fill daily map with zeroes
for (let i = actualDays-1; i >= 0; i--) {
  const d = new Date(now);
  d.setDate(d.getDate() - i);
  dailyMap[d.toISOString().split("T")[0]] = 0;
}

How UTM analytics are built

In GET /urldata/analytics/:shortid, after computing base URL stats, the backend fetches all Utm docs linked to this urlId. For each UTM it filters visithistory[] by the same date range and groups clicks by source/medium/campaign using reduce maps. Produces sorted arrays powering the breakdown charts.

// Aggregate by source
utmBreakdown.forEach(u => {
  const s = u.utmSource || "direct";
  sourceMap[s] = (sourceMap[s] || 0) + u.clicks;
});
Tracking link structure
getUtmLink() — frontend

The tracking link isn't stored in DB — built on the frontend from the base URL's shorturl and the UTM document's _id. This means the Utm document doesn't need its own short code.

getUtmLink(utm): string {
  return `${apiUrl}/go/${utm.urlId.shorturl}?utm_id=${utm._id}`;
}
Dual logging explained
Click counting behaviour

Clicking a UTM link counts as 1 click on the base URL total AND 1 click on the UTM link. If you have 100 total clicks on a URL and 60 came via UTM links, the base URL shows 100, the UTMs show 60 combined. The 60 UTM clicks are a subset of the 100 total.
